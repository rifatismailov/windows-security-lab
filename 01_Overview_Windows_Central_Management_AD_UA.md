![Сторінка 3](images/01/page_003.png)

**Microsoft SCCM**

- Орієнтований більше на робочі станції (workstations) та сервери
- Основні функції:
  - Управління конфігурацією (configuration management)
  - Управління виправленнями (patch management)
  - Розповсюдження програмного забезпечення (software distribution)
  - Розгортання операційної системи (operating system deployment)
  - Апаратна та програмна інвентаризація (hardware and software inventory)
  - Управління живленням (power management)
- Завантаження по мережі (PXE boot) і розгортання «з нуля» (bare metal imaging)

## Сторінка 4

![Сторінка 4](images/01/page_004.png)

**Секвенсор завдань SCCM (SCCM Task Sequencer)**

- Одна з найбільших переваг SCCM — можливість створювати автоматизовані процеси підготовки робочих станцій / серверів

**«Автоматизовані процеси підготовки» (provisioning)** — означає:

- Замість того щоб вручну налаштовувати кожен комп'ютер/сервер, SCCM дозволяє описати конфігурацію один раз і автоматично розгортати її на сотні/тисячі машин
- Це включає: встановлення ОС, інсталяцію програм, налаштування політик, оновлення

**Практичний приклад:**
Потрібно розгорнути 500 робочих станцій з Windows 11 + Office + антивірус. Без SCCM — це тижні ручної роботи. З SCCM — адміністратор створює task sequence (послідовність завдань), і всі машини налаштовуються автоматично за одну ніч.

**Ключові механізми SCCM для цього:**
- **OSD (OS Deployment)** — автоматична установка ОС
- **Task Sequences** — сценарії підготовки крок за кроком
- **PXE Boot** — завантаження по мережі без фізичних носіїв

## Сторінка 5

![Сторінка 5](images/01/page_005.png)

**Віртуалізація застосунків (Application Virtualization — App-V)**

- Потокове передавання застосунку (streaming application) замість його встановлення
- Весь застосунок існує всередині «бульбашки» (bubble) та передається потоком із сервера AppV
- Мінімальний вплив на ОС (OS tampering) через відсутність процесів «встановлення» (install) або «видалення» (uninstall)

**Основна ідея:**
Застосунок **не встановлюється** на комп'ютер у звичайному сенсі — він існує в ізольованому середовищі («бульбашці») і передається з сервера по мережі.

**Аналогія:** Як стрімінг відео (Netflix) — фільм не завантажується повністю, а передається потоком. App-V робить те саме із застосунками.

**Що таке «бульбашка» (bubble)?**

Ізольований контейнер, всередині якого застосунок:
- «думає», що встановлений на комп'ютері
- має свій власний реєстр (registry)
- має свої власні файли та бібліотеки
- **не взаємодіє з іншими застосунками та ОС**

**Переваги:**

| Звичайне встановлення | App-V |
|---|---|
| Змінює реєстр, системні файли | Нічого не змінює в ОС |
| Конфлікти між застосунками | Кожен в ізольованій бульбашці |
| Видалення залишає «сміття» | Видалення = просто прибрати потік |
| Треба встановлювати на кожній машині | Один сервер → всі машини |

**Практичний приклад:**
У компанії є два відділи: бухгалтерія потребує **Excel 2016**, розробники — **Excel 2021**. На одному комп'ютері дві версії конфліктують. З App-V — кожна версія у своїй бульбашці, без конфліктів.

**Ключові компоненти:**
- **App-V Server** — зберігає та роздає застосунки
- **App-V Client** — на кожній робочій станції, отримує потік
- **Sequencer** — «пакує» застосунок у формат App-V

**Головна перевага для адміністратора:** Оновив застосунок на сервері — всі користувачі автоматично отримали нову версію, без обходу кожного комп'ютера.

## Сторінка 8

![Сторінка 8](images/01/page_008.png)

**Розділ (Partition)**

Розділ, або контекст іменування (naming context), — це частина бази даних AD DS. Хоча база даних складається з одного файлу під назвою `Ntds.dit`, різні розділи містять різні дані. Наприклад, розділ схеми (schema partition) містить копію схеми Active Directory. Розділ конфігурації (configuration partition) містить об'єкти конфігурації для лісу (forest), а розділ домену (domain partition) містить користувачів, комп'ютери, групи та інші об'єкти, що належать до конкретного домену. Active Directory зберігає копії розділів на кількох контролерах домену (domain controllers) і оновлює їх через реплікацію каталогу (directory replication).

**Три основні розділи:**

**1. Schema Partition (Розділ схеми)**
- Зберігає «правила» — які об'єкти існують в AD і які атрибути вони можуть мати
- Приклад: «об'єкт користувача має поля: ім'я, пароль, email»
- **Один на весь ліс** — однаковий на всіх контролерах домену

**2. Configuration Partition (Розділ конфігурації)**
- Зберігає структуру всього лісу: сайти, служби, зв'язки між доменами
- **Один на весь ліс** — реплікується на всі контролери домену

**3. Domain Partition (Розділ домену)**
- Зберігає самі об'єкти: користувачів, комп'ютери, групи, OU
- **Унікальний для кожного домену** — DC домену A не має розділу домену B

**Схема структури Ntds.dit:**

```
┌─────────────────────────────────────────────────────┐
│                  Файл Ntds.dit                      │
│                                                     │
│  ┌─────────────────┐  ┌──────────────────────────┐  │
│  │ Schema Partition│  │ Configuration Partition  │  │
│  │                 │  │                          │  │
│  │ - типи об'єктів │  │ - структура лісу         │  │
│  │ - атрибути      │  │ - сайти                  │  │
│  │                 │  │ - зв'язки між доменами   │  │
│  └─────────────────┘  └──────────────────────────┘  │
│       (весь ліс)              (весь ліс)             │
│                                                     │
│  ┌──────────────────────────────────────────────┐   │
│  │            Domain Partition                  │   │
│  │                                              │   │
│  │  Користувачі │ Комп'ютери │ Групи │ OU       │   │
│  └──────────────────────────────────────────────┘   │
│                  (тільки цей домен)                 │
└─────────────────────────────────────────────────────┘
```

**Реплікація між контролерами домену:**

```
  DC1 (контролер 1)        DC2 (контролер 2)
  ┌──────────────┐          ┌──────────────┐
  │ Schema       │◄────────►│ Schema       │
  │ Config       │◄────────►│ Config       │
  │ Domain       │◄────────►│ Domain       │
  └──────────────┘          └──────────────┘
         │                        │
         └────────────────────────┘
              реплікація
```

Якщо один DC падає — інший має **повну копію** всіх розділів і продовжує працювати.

> **Простими словами:** Уяви книгу (Ntds.dit) з трьома розділами: «Зміст» (Schema), «Карта» (Configuration) і «Головний текст» (Domain). Кожен контролер домену має свою копію цієї книги і постійно звіряється з іншими копіями.

## Сторінка 9

![Сторінка 9](images/01/page_009.png)

**Схема (Schema)**

Схема (schema) — це набір визначень типів об'єктів і атрибутів, які використовуються для визначення об'єктів, що створюються в AD DS.

**Проста аналогія:**
Схема — це як **бланк анкети**. Анкета визначає, які поля існують (ім'я, вік, адреса). Об'єкт AD — це заповнена анкета з конкретними даними.

**Два ключових поняття схеми:**

**1. Класи (Classes)** — типи об'єктів, які можна створити в AD:
- `user` — користувач
- `computer` — комп'ютер
- `group` — група
- `organizationalUnit` — OU

**2. Атрибути (Attributes)** — властивості, які може мати клас:
- Клас `user` має атрибути: `sAMAccountName`, `mail`, `displayName`, `password` тощо
- Клас `computer` має атрибути: `dNSHostName`, `operatingSystem` тощо

**Схема:**

```
        СХЕМА (Schema)
        ┌──────────────────────────────────────┐
        │  Клас: user                          │
        │  ┌────────────────────────────────┐  │
        │  │ Атрибути:                      │  │
        │  │  - sAMAccountName (логін)      │  │
        │  │  - displayName   (ім'я)        │  │
        │  │  - mail          (email)       │  │
        │  │  - password      (пароль)      │  │
        │  │  - ...400+ атрибутів           │  │
        │  └────────────────────────────────┘  │
        │                                      │
        │  Клас: computer                      │
        │  ┌────────────────────────────────┐  │
        │  │ Атрибути:                      │  │
        │  │  - dNSHostName  (ім'я хоста)   │  │
        │  │  - operatingSystem             │  │
        │  │  - ...                         │  │
        │  └────────────────────────────────┘  │
        └──────────────────────────────────────┘
                       ▼
        РЕАЛЬНИЙ ОБ'ЄКТ (заповнена анкета)
        ┌──────────────────────────────────────┐
        │  Об'єкт типу: user                   │
        │  sAMAccountName = "john.doe"          │
        │  displayName    = "John Doe"          │
        │  mail           = "john@company.com"  │
        └──────────────────────────────────────┘
```

**Важливі факти:**
- Схема **єдина для всього лісу** — не можна мати різні схеми в різних доменах
- Схему можна **розширювати** — додавати нові атрибути або класи (наприклад, Exchange додає свої атрибути до класу `user`)
- Схему **не можна звужувати** — видалити атрибут зі схеми неможливо, лише деактивувати
- Змінювати схему може лише член групи **Schema Admins**

> **Простими словами:** Схема — це «словник» AD, який каже: «ось які типи об'єктів існують і які поля вони мають». Без схеми AD не знало б, що таке «користувач» і які дані про нього зберігати.

## Сторінка 10

![Сторінка 10](images/01/page_010.png)

**Домен (Domain)**

Домен (domain) — це логічний адміністративний контейнер для таких об'єктів, як користувачі та комп'ютери. Домен відповідає певному розділу (partition), і домени можна організовувати у відносини батько-дитина (parent-child relationships) з іншими доменами.

**Проста аналогія:**
Домен — це як **місто**. Місто має свої правила, своїх жителів (користувачів) і свою адміністрацію (контролер домену). Різні міста можуть входити до однієї області (лісу), але кожне місто керує собою самостійно.

**Що зберігається в домені:**
- Облікові записи користувачів
- Облікові записи комп'ютерів
- Групи
- Групові політики (GPO)
- Організаційні підрозділи (OU)

**Ключові характеристики:**
- Кожен домен має своє **DNS-ім'я** (наприклад, `company.com`)
- Кожен домен має свій власний **Domain Partition** у базі Ntds.dit
- Домен — це **межа реплікації**: дані домену реплікуються між DC цього домену, але не між доменами
- Домен — це **межа адміністрування**: адміністратор одного домену не керує іншим доменом автоматично

**Відносини батько-дитина (Parent-Child):**

```
        company.com          ← батьківський домен (parent)
        ┌─────────────┐
        │  DC          │
        │  Користувачі │
        │  Комп'ютери  │
        └──────┬───────┘
               │
       ┌───────┴────────┐
       ▼                ▼
  kyiv.company.com   lviv.company.com   ← дочірні домени (child)
  ┌────────────┐     ┌────────────┐
  │ DC         │     │ DC         │
  │ Київські   │     │ Львівські  │
  │ юзери      │     │ юзери      │
  └────────────┘     └────────────┘
```

**Домен і розділ (partition):**

```
  Домен: company.com
  ┌──────────────────────────────┐
  │  Ntds.dit                    │
  │  ┌──────────────────────┐    │
  │  │ Domain Partition     │    │
  │  │ (тільки company.com) │    │
  │  └──────────────────────┘    │
  │  ┌──────────────┐            │
  │  │ Schema       │ (ліс)      │
  │  └──────────────┘            │
  │  ┌──────────────┐            │
  │  │ Config       │ (ліс)      │
  │  └──────────────┘            │
  └──────────────────────────────┘
```

> **Простими словами:** Домен — це окрема «адміністративна зона» з власними об'єктами та правилами. Домени можна вкладати один в одний як матрьошки — батьківський домен містить дочірні.

## Сторінка 11

![Сторінка 11](images/01/page_011.png)

**Дерево доменів (Domain Tree)**

Дерево доменів — це ієрархічна сукупність доменів, що мають спільний кореневий домен (root domain) та суміжний простір імен DNS (contiguous Domain Name System namespace).

**Проста аналогія:**
Дерево доменів — як **адреса в місті**: `вулиця.район.місто`. Кожна частина є дочірньою до попередньої, і всі вони пов'язані спільною назвою.

**Що таке «суміжний простір імен»?**
Всі домени в дереві мають DNS-імена, що є продовженням кореневого:

```
        company.com              ← кореневий домен дерева
        ┌───────────┐
        │           │
        ▼           ▼
  kyiv.company.com  lviv.company.com   ← дочірні домени
        │
        ▼
  sales.kyiv.company.com              ← онук кореневого
```

Усі імена містять `company.com` — це і є «суміжний простір імен».

**Між доменами в дереві автоматично створюються двосторонні транзитивні довіри:**

```
  company.com  ◄──довіра──►  kyiv.company.com
  kyiv.company.com  ◄──довіра──►  sales.kyiv.company.com
  ∴ company.com також довіряє sales.kyiv.company.com (транзитивно)
```

> **Простими словами:** Дерево доменів — це сім'я доменів зі спільним «прізвищем» (DNS-суфіксом). Батьки і діти автоматично довіряють одне одному.

## Сторінка 12

![Сторінка 12](images/01/page_012.png)

**Ліс (Forest)**

Ліс — це сукупність одного або кількох доменів, що мають спільний корінь AD DS (AD DS root), спільну схему (schema) та спільний глобальний каталог (global catalog).

**Проста аналогія:**
Ліс — це **країна**, що складається з міст (доменів). Міста можуть бути об'єднані в регіони (дерева доменів), але всі вони живуть за спільними законами (схема) і мають спільну карту (глобальний каталог).

**Ліс може містити кілька дерев з різними DNS-іменами:**

```
                    ЛІС (Forest)
    ┌───────────────────────────────────────────┐
    │                                           │
    │   Дерево 1           Дерево 2             │
    │   company.com        fabrikam.com         │
    │   ┌──────────┐       ┌──────────┐         │
    │   │          │       │          │         │
    │   ▼          ▼       ▼          ▼         │
    │ kyiv.    lviv.    berlin.    paris.        │
    │ company  company  fabrikam   fabrikam      │
    │ .com     .com     .com       .com          │
    │                                           │
    │  Спільне для всього лісу:                 │
    │  ✔ Schema Partition                       │
    │  ✔ Configuration Partition                │
    │  ✔ Global Catalog                         │
    └───────────────────────────────────────────┘
```

**Ліс = межа безпеки:**
- Користувачі з іншого лісу **не мають доступу** за замовчуванням
- Всередині лісу всі домени автоматично довіряють одне одному
- Між лісами довіру треба налаштовувати вручну

> **Простими словами:** Ліс — найбільший контейнер в AD. Один ліс = одна організація (або кілька, якщо є міжлісові довіри). Все всередині лісу підкоряється спільним правилам (схемі).

## Сторінка 13

![Сторінка 13](images/01/page_013.png)

**Організаційний підрозділ (Organizational Unit — OU)**

Організаційний підрозділ (OU) — це контейнерний об'єкт для користувачів, груп і комп'ютерів, який забезпечує основу для делегування адміністративних прав (administrative rights) і управління шляхом прив'язки об'єктів групової політики (Group Policy Objects — GPOs).

**Проста аналогія:**
OU — це як **папка на диску**. Папки можна вкладати одна в одну, застосовувати до них права доступу та правила. Об'єкти всередині папки успадковують її налаштування.

**Для чого використовується OU:**

1. **Групування об'єктів** — зібрати всіх користувачів бухгалтерії в одну OU
2. **Застосування GPO** — призначити політики лише для цієї групи
3. **Делегування прав** — дати локальному адміну права лише на свою OU

**Схема структури OU:**

```
  company.com (домен)
  ├── OU: Kyiv
  │   ├── OU: IT
  │   │   ├── user: john.doe
  │   │   └── computer: PC-IT-01
  │   └── OU: Finance
  │       ├── user: jane.smith
  │       └── GPO: Фінансові обмеження ──► застосовується до всіх у Finance
  └── OU: Lviv
      └── OU: HR
          └── user: bob.johnson
```

**OU vs Контейнер — головна різниця:**

| OU | Контейнер (Container) |
|---|---|
| Можна прив'язати GPO | **Не можна** прив'язати GPO |
| Можна делегувати права | Обмежене делегування |
| Створюється адміністратором | Вбудований в AD (Users, Computers) |

> **Простими словами:** OU — це «розумна папка» для об'єктів AD, до якої можна застосовувати політики та делегувати управління.

## Сторінка 14

![Сторінка 14](images/01/page_014.png)

**Контейнер (Container)**

Контейнер (container) — це об'єкт, що забезпечує організаційну структуру в AD DS. Можна використовувати стандартні контейнери або створювати власні. До контейнерів не можна прив'язувати об'єкти групової політики (GPOs).

**Проста аналогія:**
Контейнер — це як **системна папка Windows** (наприклад, `C:\Windows\System32`). Вона існує за замовчуванням, має фіксоване призначення, і ти не можеш змінити її поведінку так само гнучко, як звичайну папку.

**Вбудовані контейнери в AD:**

```
  company.com (домен)
  ├── [Контейнер] Builtin     ← вбудовані групи (Administrators, Users...)
  ├── [Контейнер] Computers   ← нові комп'ютери потрапляють сюди за замовчуванням
  ├── [Контейнер] Users       ← нові користувачі потрапляють сюди за замовчуванням
  ├── [Контейнер] ForeignSecurityPrincipals
  └── [OU] Domain Controllers ← це OU, не контейнер!
```

**Ключова проблема контейнерів:**
Нові комп'ютери автоматично потрапляють до контейнера `Computers`, а не до OU. Це означає, що GPO до них **не застосовуються**, поки адміністратор вручну не перемістить їх до потрібної OU.

**Рішення:** Змінити стандартне розміщення нових комп'ютерів командою:
```
redircmp "OU=Workstations,DC=company,DC=com"
```

> **Простими словами:** Контейнер — це «проста папка» без можливості застосовувати GPO. OU — «розумна папка» з повним набором можливостей. Завжди краще використовувати OU замість контейнерів.

> ❓ [Питання: якщо OU краще — навіщо взагалі існують контейнери?](01_FAQ.md#-якщо-ou-краще-за-контейнер--навіщо-взагалі-існують-контейнери)

## Сторінка 16

![Сторінка 16](images/01/page_016.png)

**Контролер домену (Domain Controller)**

Контролер домену містить копію бази даних AD DS. Для більшості операцій кожен контролер домену може обробляти зміни та реплікувати їх на всі інші контролери домену в домені.

**Проста аналогія:**
DC — це як **сервер банку**, що зберігає всі рахунки клієнтів. Коли ти входиш у систему — DC перевіряє твій «рахунок» (обліковий запис) і підтверджує особу.

**Що робить DC:**
- **Автентифікація** — перевіряє логін і пароль при вході
- **Авторизація** — визначає, до яких ресурсів є доступ
- **Реплікація** — синхронізує зміни з іншими DC
- **DNS** — зазвичай також є DNS-сервером для домену

**Мультимайстерна реплікація (Multi-master):**

```
     DC1                DC2                DC3
  ┌───────┐          ┌───────┐          ┌───────┐
  │ Ntds  │◄────────►│ Ntds  │◄────────►│ Ntds  │
  │ .dit  │          │ .dit  │          │ .dit  │
  └───────┘          └───────┘          └───────┘
      ▲                                     │
      └─────────────────────────────────────┘

  Зміни можна вносити на БУДЬ-ЯКИЙ DC → реплікується на всі інші
```

**Чому потрібно мінімум 2 DC:**

```
  Один DC (погано):          Два DC (добре):
  ┌───────┐                  ┌───────┐    ┌───────┐
  │  DC1  │  ← впав!         │  DC1  │    │  DC2  │
  │       │                  │       │    │       │
  └───────┘                  └───┬───┘    └───┬───┘
  Весь домен недоступний!        └────────────┘
                             DC1 впав → DC2 продовжує
```

> **Простими словами:** DC — це «серце» домену. Він знає всіх користувачів і комп'ютери, перевіряє паролі і роздає права. Без DC домен не працює, тому їх завжди має бути мінімум два.

> ❓ [Питання: на одному Windows Server може бути кілька DC чи кожен DC — окремий сервер?](01_FAQ.md#-на-одному-windows-server-може-бути-кілька-dc-dc1-dc2-чи-кожен-dc--окремий-сервер)

## Сторінка 17

![Сторінка 17](images/01/page_017.png)

**Сховище даних (Data Store)**

Копія сховища даних (data store) існує на кожному контролері домену. База даних AD DS використовує технологію баз даних Microsoft Jet і зберігає інформацію каталогу у файлі `Ntds.dit` та пов'язаних журнальних файлах. За замовчуванням ці файли зберігаються в папці `C:\Windows\NTDS`.

**Що знаходиться в `C:\Windows\NTDS`:**

```
  C:\Windows\NTDS\
  ├── Ntds.dit       ← головний файл бази даних (всі об'єкти AD)
  ├── edb.log        ← журнал транзакцій (останні зміни)
  ├── edb00001.log   ← старіші журнали транзакцій
  ├── edb.chk        ← checkpoint-файл (що вже записано в .dit)
  └── res1.log       ← резервний простір на диску
```

**Як працює запис змін (для надійності):**

```
  1. Адмін змінює пароль користувача
           ▼
  2. Зміна записується в журнал (edb.log) — швидко!
           ▼
  3. Зміна застосовується до Ntds.dit — трохи пізніше
           ▼
  4. Checkpoint оновлюється — підтверджує запис у .dit
```

Якщо DC вимикається до кроку 3 — при наступному запуску журнал «довідує» базу до актуального стану. Дані не губляться.

**Microsoft Jet — що це?**
Та сама технологія бази даних, що використовується в Microsoft Access. Але для AD вона суттєво доопрацьована і оптимізована для мільйонів об'єктів.

> **Простими словами:** `Ntds.dit` — це «серце» DC, файл де живуть всі користувачі, комп'ютери і групи. Журнали (`edb.log`) — страховка на випадок раптового вимкнення.

> ❓ [Питання: Data Store — це сховище файлів на Windows Server? Хто має доступ?](01_FAQ.md#-data-store--це-сховище-файлів-cwindowsntds-на-windows-server-хто-має-до-нього-доступ)

## Сторінка 18

![Сторінка 18](images/01/page_018.png)

**Сервер глобального каталогу (Global Catalog Server)**

Сервер глобального каталогу — це контролер домену, який розміщує глобальний каталог (global catalog) — частковий, доступний лише для читання примірник усіх об'єктів у мультидоменному лісі (multiple-domain forest). Глобальний каталог прискорює пошук об'єктів, які можуть зберігатися на контролерах домену в іншому домені лісу.

**Проста аналогія:**
GC — це як **загальний телефонний довідник** для всієї компанії. Кожен офіс (домен) має повний список своїх співробітників. GC — це скорочений список **усіх** співробітників з усіх офісів, де є лише базова інформація (ім'я, email, відділ).

**Що зберігає GC:**

```
  Звичайний DC (домен A):        Global Catalog Server:
  ┌──────────────────────┐       ┌────────────────────────────────┐
  │ Всі атрибути         │       │ Часткові дані з УСІХ доменів   │
  │ користувачів домену A│       │                                │
  │ (~400 атрибутів)     │       │ Домен A: name, mail, memberOf  │
  └──────────────────────┘       │ Домен B: name, mail, memberOf  │
                                 │ Домен C: name, mail, memberOf  │
  Знає все про своїх,            └────────────────────────────────┘
  нічого про чужих               Знає трохи про всіх у лісі
```

**Навіщо потрібен GC:**

```
  Сценарій: Користувач з домену A шукає колегу з домену B

  БЕЗ GC:                        З GC:
  Запит → DC домену A             Запит → GC сервер
  «Не знаю» → запит до DC B      «Ось результат!» ✔
  → очікування → відповідь
  (повільно, складно)             (швидко, один запит)
```

**GC також потрібен для входу в систему:**
При вході Windows перевіряє членство в універсальних групах через GC. Якщо GC недоступний — вхід може бути заблокований (крім адміністраторів домену).

> **Простими словами:** GC — це «жовті сторінки» всього лісу AD. Замість того щоб запитувати кожен домен окремо, можна запитати GC і отримати базову інформацію про будь-який об'єкт лісу.

## Сторінка 19

![Сторінка 19](images/01/page_019.png)

**Контролер домену лише для читання (Read-Only Domain Controller — RODC)**

RODC — це спеціальна, доступна лише для читання інсталяція AD DS. RODC поширені у філіях, де фізична безпека не є оптимальною, ІТ-підтримка менш розвинена, ніж у головних корпоративних центрах, або де бізнес-застосунки (line-of-business applications) мають запускатися на контролері домену.

**Проста аналогія:**
RODC — як **ксерокопія документа**. Ти можеш її читати та використовувати, але записати зміни в неї не можеш — зміни вносяться тільки в оригінал (звичайний DC) і потім копіюються на RODC.

**Де використовується RODC:**

```
  Головний офіс (Київ)           Філія (невелике місто)
  ┌─────────────────┐            ┌─────────────────┐
  │  DC (повний)    │            │  RODC           │
  │  ✔ Читання      │◄──────────►│  ✔ Читання      │
  │  ✔ Запис        │  реплікація│  ✗ Запис        │
  │  Захищений      │            │  Фізично        │
  │  серверний зал  │            │  незахищене     │
  └─────────────────┘            │  приміщення     │
                                 └─────────────────┘
```

**Ключові переваги RODC:**

| Проблема філії | Рішення RODC |
|---|---|
| Сервер може бути вкрадений | База даних лише для читання — зловмисник не зможе змінити AD |
| Немає кваліфікованого IT | RODC може адмініструвати локальний співробітник без прав Domain Admin |
| Повільний канал до головного офісу | RODC кешує паролі локальних користувачів → вхід без запиту до головного DC |
| Небезпечні застосунки на DC | Компрометація RODC не дає змоги змінити весь AD |

**Password Replication Policy (PRP):**
RODC може кешувати паролі лише дозволених користувачів:
```
  Дозволено кешувати:  users у групі "Kyiv-Branch-Users"
  Заборонено:          Domain Admins, Schema Admins (завжди!)
```

> **Простими словами:** RODC — безпечна версія DC для місць, де сервер може опинитись у чужих руках. Навіть якщо його вкрадуть — зловмисник отримає лише копію для читання, без можливості змінити AD.

## Сторінка 20

![Сторінка 20](images/01/page_020.png)

**Сайт (Site)**

Сайт — це контейнер для об'єктів AD DS, таких як комп'ютери та служби, прив'язані до конкретного фізичного місця розташування. Це відрізняється від домену (domain), який представляє логічну структуру об'єктів, таких як користувачі та групи, на додаток до комп'ютерів.

**Проста аналогія:**
Сайт — це **фізичний офіс**. Домен — це **відділ компанії**. Один відділ може мати співробітників у різних офісах. Один офіс може містити співробітників з різних відділів.

**Сайт vs Домен — ключова різниця:**

```
  ДОМЕН (логічна структура)      САЙТ (фізична структура)
  ┌──────────────────────┐       ┌──────────────────────┐
  │ company.com          │       │ Сайт: Київ           │
  │ - Бухгалтерія (OU)   │       │ IP: 10.1.0.0/24      │
  │ - IT відділ (OU)     │       │ DC1, DC2             │
  │ - HR (OU)            │       │ 200 комп'ютерів      │
  └──────────────────────┘       └──────────────────────┘
  Хто і як організований         Де фізично знаходиться
```

**Навіщо потрібні сайти:**

```
       Київ (сайт 1)                  Львів (сайт 2)
       10.1.0.0/24                    10.2.0.0/24
    ┌──────────────┐               ┌──────────────┐
    │  DC1  DC2    │               │    DC3       │
    │              │               │              │
    │ 200 ПК       │◄──WAN-канал──►│ 50 ПК        │
    └──────────────┘   (повільний) └──────────────┘

  Реплікація всередині сайту:  швидка, часта (кожні 15 сек)
  Реплікація між сайтами:      стиснена, рідша (за розкладом)
```

**Головне призначення сайтів:**
1. **Оптимізація реплікації** — між офісами реплікувати лише стиснені дані за розкладом, а не постійно
2. **Оптимізація входу** — комп'ютер у Києві автентифікується на київському DC, а не їде через WAN до Львова
3. **Розгортання ПЗ через GPO** — доставляти установники з найближчого сервера

> **Простими словами:** Сайт = фізичне місце з хорошим мережевим зв'язком. AD використовує сайти, щоб не гнати зайвий трафік через повільні WAN-канали між офісами.

> ❓ [Питання: Сайт (Site) в AD — це веб-сайт чи щось інше?](01_FAQ.md#-сайт-site-в-ad--це-веб-сайт-чи-щось-інше)

## Сторінка 21

![Сторінка 21](images/01/page_021.png)

**Підмережа (Subnet)**

Підмережа — це частина IP-адрес мережі організації, призначених комп'ютерам у певному сайті (site). Один сайт може мати більше однієї підмережі.

**Проста аналогія:**
Якщо Сайт — це місто, то Підмережа — це **поштовий індекс** цього міста. За IP-адресою комп'ютера AD визначає до якого сайту він належить.

**Як AD визначає сайт комп'ютера:**

```
  Комп'ютер має IP: 10.1.5.25
           ▼
  AD перевіряє список підмереж:
  ┌─────────────────────────────────────┐
  │ 10.1.0.0/24  → Сайт: Kyiv-Site     │
  │ 10.2.0.0/24  → Сайт: Lviv-Site     │
  │ 10.3.0.0/24  → Сайт: Odesa-Site    │
  └─────────────────────────────────────┘
           ▼
  10.1.5.25 належить до 10.1.0.0/24
           ▼
  Комп'ютер у Kyiv-Site → автентифікація через київський DC
```

**Один сайт може мати кілька підмереж:**

```
  Сайт: Kyiv-Site
  ┌──────────────────────────────────┐
  │  Підмережа 1: 10.1.0.0/24       │  ← поверх 1-5
  │  Підмережа 2: 10.1.1.0/24       │  ← поверх 6-10
  │  Підмережа 3: 192.168.1.0/24    │  ← Wi-Fi мережа
  └──────────────────────────────────┘
  Всі в одному будинку → один сайт
```

**Що станеться якщо підмережу не додати до сайту:**

```
  Новий офіс, підмережа не зареєстрована в AD
           ▼
  Комп'ютер не знає свого сайту
           ▼
  AD обирає DC випадково — можливо за океаном!
           ▼
  Вхід в систему — повільно, трафік іде через WAN
```

> **Простими словами:** Підмережа — це «поштовий індекс» для AD. Вона каже системі: «цей комп'ютер знаходиться ось тут» і AD направляє його до найближчого DC.

## Сторінка 22

![Сторінка 22](images/01/page_022.png)

**Об'єкти користувачів в AD DS (User Objects in AD DS)**

- Обліковий запис користувача (user account) надає доступ до мережевих ресурсів, де користувач може пройти автентифікацію (authenticate) в AD DS для доступу до ресурсів
- Обліковий запис користувача включає:
  - ім'я користувача (username)
  - пароль користувача (user password)
  - членство в групах (group memberships)
  - багато інших організаційних інформаційних полів

**Проста аналогія:**
Обліковий запис користувача — це **перепустка в офіс**. Перепустка має фото (ім'я), PIN-код (пароль) і список кімнат куди можна входити (членство в групах).

**Що реально зберігається в обліковому записі:**

```
  Об'єкт користувача в AD
  ┌─────────────────────────────────────────────┐
  │ Основні атрибути:                           │
  │   sAMAccountName : john.doe                 │  ← логін (DOMAIN\john.doe)
  │   userPrincipalName: john.doe@company.com   │  ← UPN (для входу як email)
  │   displayName    : John Doe                 │  ← відображуване ім'я
  │   mail           : john.doe@company.com     │  ← email
  │   unicodePwd     : [хеш пароля]             │  ← пароль (зберігається як хеш!)
  │                                             │
  │ Членство в групах:                          │
  │   memberOf: IT-Staff, VPN-Users, Office365  │
  │                                             │
  │ Службові атрибути:                          │
  │   accountExpires : 31.12.2026               │  ← коли закінчується доступ
  │   lastLogon      : 2026-05-24 09:15         │  ← останній вхід
  │   pwdLastSet     : 2026-03-01               │  ← коли змінювали пароль
  │   userAccountControl: NORMAL_ACCOUNT        │  ← статус (активний/заблокований)
  └─────────────────────────────────────────────┘
```

**Як працює автентифікація:**

```
  Користувач вводить логін + пароль
           ▼
  Windows надсилає запит до DC (Kerberos)
           ▼
  DC перевіряє хеш пароля в Ntds.dit
           ▼
  Пароль вірний → DC видає квиток (Ticket)
           ▼
  Квиток пред'являється ресурсу (файловий сервер, пошта...)
           ▼
  Доступ надано або відмовлено на основі груп
```

> **Пароль НІКОЛИ не зберігається у відкритому вигляді** — тільки його математичний відбиток (хеш). DC порівнює хеш введеного пароля з хешем у базі.

## Сторінка 23

![Сторінка 23](images/01/page_023.png)

**Облікові записи керованих служб (Managed Service Accounts — MSA)**

- Багато застосунків містять служби (services), що виконуються у фоновому режимі та забезпечують взаємодію з застосунком
- Часто застосункам не потрібна взаємодія з користувачем, але може знадобитися доступ до додаткових ресурсів AD DS (мережеві папки, файли та теки на сервері)
- MSA можуть бути локальними для комп'ютера (Local Service, Network Service, Local System) або доменними (domain based)

**Проста аналогія:**
MSA — це **службовий пропуск для робота**. Людині потрібен особистий пропуск з PIN-кодом. Роботу (службі) теж потрібен пропуск, але він не може вводити PIN — тому його пропуск оновлюється автоматично.

**Чому потрібні MSA — проблема зі звичайними обліковими записами:**

```
  Звичайна служба (SQL Server, IIS...) запускається від імені:

  Варіант А: Локальний системний акаунт
  ┌─────────────────────────────────────┐
  │ Local System / Network Service      │
  │ ✔ Не потрібен пароль                │
  │ ✗ Немає доступу до мережевих ресурсів│
  └─────────────────────────────────────┘

  Варіант Б: Звичайний доменний акаунт
  ┌─────────────────────────────────────┐
  │ DOMAIN\svc-sqlserver                │
  │ ✔ Є доступ до мережевих ресурсів    │
  │ ✗ Пароль треба оновлювати вручну    │
  │ ✗ Пароль прострочився → служба впала│
  │ ✗ Важко відстежити що це службовий  │
  └─────────────────────────────────────┘
```

**MSA вирішує ці проблеми:**

```
  MSA (Managed Service Account)
  ┌─────────────────────────────────────┐
  │ DOMAIN\SQLServer$                   │  ← $ в кінці = це MSA
  │ ✔ Є доступ до мережевих ресурсів    │
  │ ✔ Пароль змінюється автоматично     │  ← кожні 30 днів, AD сам міняє
  │ ✔ SPN керується автоматично         │
  │ ✔ Одразу видно що це службовий акаунт│
  └─────────────────────────────────────┘
```

**Типи службових акаунтів:**

```
  ┌──────────────┬───────────────────────────────────────────────┐
  │ Local System │ Повний доступ до локального ПК, немає мережі  │
  │ Network Svc  │ Обмежений локальний + базовий мережевий доступ│
  │ Local Svc    │ Мінімальний локальний, немає мережі           │
  │ MSA          │ Доменний, автопароль, один сервер             │
  │ gMSA         │ Доменний, автопароль, кілька серверів         │
  └──────────────┴───────────────────────────────────────────────┘
```

## Сторінка 24

![Сторінка 24](images/01/page_024.png)

**Доменний обліковий запис проти MSA (Domain Based Account vs MSA)**

**Недоліки доменного облікового запису:**
- Може знадобитися додаткове адміністративне зусилля для безпечного управління паролем
- Може бути важко визначити, чи використовується доменний обліковий запис як службовий
- Може знадобитися додаткове адміністративне зусилля для управління іменем принципала служби (SPN — Service Principal Name)

**Переваги MSA:**
- Спрощене управління паролями (password management)
- Спрощене управління SPN

**Порівняльна таблиця:**

```
  ┌────────────────────────┬──────────────────┬──────────────────┐
  │ Характеристика         │ Доменний акаунт  │ MSA              │
  ├────────────────────────┼──────────────────┼──────────────────┤
  │ Пароль                 │ Вручну, може     │ Автоматично      │
  │                        │ прострочитись    │ кожні 30 днів    │
  ├────────────────────────┼──────────────────┼──────────────────┤
  │ Вхід як користувач     │ Може (небезпечно)│ Неможливо        │
  ├────────────────────────┼──────────────────┼──────────────────┤
  │ SPN управління         │ Вручну           │ Автоматично      │
  ├────────────────────────┼──────────────────┼──────────────────┤
  │ Видно що це служба     │ Не завжди        │ Так ($ в кінці)  │
  ├────────────────────────┼──────────────────┼──────────────────┤
  │ Кілька серверів        │ Так              │ Ні (тільки 1)    │
  └────────────────────────┴──────────────────┴──────────────────┘
```

**Що таке SPN (Service Principal Name):**
SPN — це унікальний ідентифікатор служби в домені для Kerberos автентифікації. Приклад: `MSSQLSvc/sqlserver.company.com:1433`. Без правильного SPN Kerberos не зможе автентифікувати клієнтів до служби.

## Сторінка 25

![Сторінка 25](images/01/page_025.png)

**Групові облікові записи керованих служб (Group Managed Service Accounts — gMSA)**

Групові облікові записи керованих служб дозволяють використовувати MSA на кількох серверах домену (наприклад, кластери балансування навантаження або сервери IIS).

Для підтримки gMSA необхідно створити кореневий ключ KDS (Key Distribution Services) на контролері домену:

```
Add-KdsRootKey -EffectiveImmidiately
New-ADServiceAccount -Name SQLfarm –PrincipalsAllowedToRetrieveManagedPassword SERVER1,SERVER2,SERVER3
```

**Різниця MSA vs gMSA:**

```
  MSA — один сервер:                gMSA — кілька серверів:

  ┌──────────┐                      ┌──────────┐
  │ SERVER1  │                      │ SERVER1  │◄──┐
  │ MSA:     │                      └──────────┘   │
  │ svc-app$ │                      ┌──────────┐   │  gMSA: SQLfarm$
  └──────────┘                      │ SERVER2  │◄──┤  (один акаунт
                                    └──────────┘   │   для всіх)
  ┌──────────┐                      ┌──────────┐   │
  │ SERVER2  │                      │ SERVER3  │◄──┘
  │ MSA:     │  ← потрібен          └──────────┘
  │ svc-app2$│    окремий MSA!
  └──────────┘
```

**Де використовується gMSA:**
- **IIS ферма** — кілька веб-серверів з однаковим застосунком
- **SQL кластер** — кілька вузлів SQL Server
- **Load Balancer** — сервери за балансувальником навантаження

**Як працює автоматична зміна пароля gMSA:**

```
  KDS Root Key (на DC)
  ↓ генерує пароль
  Сервери SERVER1, SERVER2, SERVER3
  → самі забирають поточний пароль у DC
  → синхронно використовують той самий пароль
  → через 30 днів DC генерує новий → сервери самі оновлюються
```

> **Простими словами:** gMSA = MSA для цілої «ферми» серверів. Один службовий акаунт, один автоматичний пароль — на всіх серверах одночасно.

> ❓ [Питання: яка реальна ціль MSA і як він працює відносно додатку та SQL Server?](01_FAQ.md#-яка-реальна-ціль-msa-і-які-роботи-він-виконує)

## Сторінка 26

![Сторінка 26](images/01/page_026.png)

**Об'єкти груп (Group Objects)**

- Групи дозволяють управляти дозволами для набору об'єктів (користувачі, комп'ютери)
- Група може бути:
  - **Група безпеки (Security group)** — група з підтримкою безпеки для призначення дозволів на різні ресурси. Групи безпеки можна використовувати для управління списками контролю доступу (ACL — Access Control Lists)
  - **Група розповсюдження (Distribution group)** — поштові застосунки використовують групи розповсюдження, які не мають підтримки безпеки. Групи безпеки можна використовувати як групи розповсюдження для поштових застосунків

**Проста аналогія:**
Група — це як **відділ у компанії**. Замість того щоб давати доступ до папки кожному співробітнику окремо — даєш доступ усьому відділу. Новий співробітник автоматично отримує всі права відділу при вступі до групи.

**Навіщо групи — без них і з ними:**

```
  БЕЗ ГРУП (погано):              З ГРУПАМИ (добре):
  Папка "Бухгалтерія"             Папка "Бухгалтерія"
  ├── Доступ: Іван ✔              └── Доступ: Група "Finance" ✔
  ├── Доступ: Марія ✔                 ├── Іван
  ├── Доступ: Петро ✔                 ├── Марія
  └── Доступ: ...50 осіб             └── Петро... (50 осіб)

  Новий співробітник?              Новий співробітник?
  Треба додати до 50 папок!        Додай до групи → все готово!
```

**Security Group vs Distribution Group:**

```
  ┌─────────────────────┬──────────────────────────────────────┐
  │ Security Group      │ Distribution Group                   │
  ├─────────────────────┼──────────────────────────────────────┤
  │ ✔ Дозволи на папки  │ ✗ Не можна призначати дозволи        │
  │ ✔ Права в системі   │ ✗ Немає підтримки безпеки (ACL)      │
  │ ✔ Можна як поштовий │ ✔ Тільки для розсилки пошти          │
  │   список теж        │                                      │
  └─────────────────────┴──────────────────────────────────────┘
```

> **Порада:** Завжди створюй Security Groups — вони вміють все що Distribution Group + ще й безпеку. Distribution Group має сенс лише якщо спеціально потрібен список без жодних прав безпеки.

## Сторінка 27

![Сторінка 27](images/01/page_027.png)

**Область дії груп (Group Scope)**

Область дії групи (group scope) визначає діапазон можливостей, дозволів та членство в групі. Підтримується 4 типи областей:

- Локальна (Local)
- Локальна домену (Domain-local)
- Глобальна (Global)
- Універсальна (Universal)

**Чому взагалі є різні scope — навіщо ця складність?**
Scope вирішує два питання: **хто може бути членом групи** і **де можна призначати дозволи цієї групи**. Це оптимізація для великих мультидоменних середовищ.

**Загальна схема всіх чотирьох типів:**

```
                        ЛІС AD
   ┌─────────────────────────────────────────────────────┐
   │  Домен A                    Домен B                 │
   │  ┌──────────────────┐       ┌──────────────────┐    │
   │  │                  │       │                  │    │
   │  │  [Local]         │       │                  │    │
   │  │  Члени: будь-хто │       │                  │    │
   │  │  Дозволи: тільки │       │                  │    │
   │  │  на цьому ПК     │       │                  │    │
   │  │                  │       │                  │    │
   │  │  [Domain-Local]  │       │                  │    │
   │  │  Члени: будь-хто │       │                  │    │
   │  │  з лісу          │       │                  │    │
   │  │  Дозволи: тільки │       │                  │    │
   │  │  в домені A      │       │                  │    │
   │  │                  │       │                  │    │
   │  │  [Global]        │──────►│  [Global]        │    │
   │  │  Члени: тільки   │       │  може бути       │    │
   │  │  з домену A      │       │  членом тут      │    │
   │  │  Дозволи: всюди  │       │                  │    │
   │  │  в лісі          │       │                  │    │
   │  └──────────────────┘       └──────────────────┘    │
   │                                                     │
   │  [Universal] — члени з будь-якого домену лісу       │
   │               дозволи — будь-де в лісі              │
   └─────────────────────────────────────────────────────┘
```

> Детальніше кожен тип розібрано на наступних сторінках (28–31).

## Сторінка 28

![Сторінка 28](images/01/page_028.png)

**Область дії груп — локальна (Group Scope — Local)**

Локальна — цей тип групи можна використовувати для автономних серверів або робочих станцій, на серверах-членах домену, що не є контролерами домену, або на робочих станціях-членах домену. Локальні групи доступні лише на тому комп'ютері, де вони існують. Важливі характеристики:

- Можна призначати можливості та дозволи лише для локальних ресурсів (на локальному комп'ютері)
- Членами можуть бути об'єкти з будь-якого місця в лісі AD DS

**Проста аналогія:**
Local Group — це як **список доступу до конкретної кімнати в будівлі**. Список існує тільки на дверях тієї кімнати. До списку можна додати будь-кого з будівлі (лісу), але ключ діє лише від цієї кімнати.

```
  PC-FILESERVER (один конкретний комп'ютер)
  ┌─────────────────────────────────────────┐
  │  Local Group: "Backup Operators"        │
  │                                         │
  │  Члени (з будь-якого місця в лісі):     │
  │   ✔ john.doe (Домен A)                  │
  │   ✔ maria.smith (Домен B)               │
  │   ✔ Global Group "IT-Admins" (Домен A)  │
  │                                         │
  │  Дозволи діють ТІЛЬКИ на цьому ПК:      │
  │   ✔ Доступ до C:\Backup                 │
  │   ✗ Не діє на FILE-SERVER-2             │
  └─────────────────────────────────────────┘
```

> **Простими словами:** Local Group існує і діє **тільки на одному конкретному комп'ютері**. Щоб дати ті самі права на іншому ПК — треба створити таку саму групу там.

## Сторінка 29

![Сторінка 29](images/01/page_029.png)

**Область дії груп — локальна домену (Group Scope — Domain-Local)**

Цей тип групи призначений насамперед для управління доступом до ресурсів або призначення прав та обов'язків адміністрування. Групи з областю дії домену існують на контролерах домену, і область дії груп обмежена доменом, в якому вони перебувають. Важливі характеристики:

- Можливості та дозволи можна призначати лише для ресурсів локального домену — тобто на всіх комп'ютерах у локальному домені
- Членами можуть бути об'єкти з будь-якого місця в лісі

**Проста аналогія:**
Domain-Local Group — це як **список доступу до цілого поверху будівлі**. До списку можна додати будь-кого з усієї будівлі (лісу), але ключ діє тільки на цьому поверсі (в цьому домені).

```
         ЛІС AD
  ┌────────────────────────────────────┐
  │  Домен A          Домен B          │
  │  ┌────────────┐   ┌────────────┐   │
  │  │DL Group:   │   │            │   │
  │  │"Finance-RW"│   │            │   │
  │  │            │   │            │   │
  │  │Члени:      │   │            │   │
  │  │✔ user з A  │   │            │   │
  │  │✔ user з B ─┼───┘ (можна!)  │   │
  │  │✔ Global    │               │   │
  │  │  Group з B │               │   │
  │  │            │               │   │
  │  │Дозволи:    │               │   │
  │  │✔ ресурси A │               │   │
  │  │✗ ресурси B │               │   │
  │  └────────────┘               │   │
  └────────────────────────────────┘
```

**Domain-Local Group використовують для призначення доступу до ресурсів**, а Global Group — для об'єднання користувачів. Разом вони утворюють стратегію AGDLP:

```
  A  → Account (обліковий запис користувача)
  G  → Global Group (об'єднує користувачів домену)
  DL → Domain-Local Group (дає доступ до ресурсу)
  P  → Permission (дозвіл на ресурс)

  Іван → Global "IT-Kyiv" → Domain-Local "FileShare-RW" → Доступ до папки
```

> **Простими словами:** Domain-Local Group — «воротар» конкретного ресурсу в домені. Вона каже «хто має доступ до цього ресурсу» і може приймати членів з усього лісу.

## Сторінка 30

![Сторінка 30](images/01/page_030.png)

**Область дії груп — глобальна (Group Scope — Global)**

Цю групу можна використовувати для об'єднання користувачів зі схожими характеристиками — наприклад, для посилання на користувачів, що належать до певного відділу або географічного розташування. Характеристики глобальних груп:

- Можна призначати можливості та дозволи будь-де в лісі
- Членами можуть бути лише об'єкти з локального домену: користувачі, комп'ютери та глобальні групи з локального домену

**Проста аналогія:**
Global Group — це як **відділ компанії**. У відділ входять тільки співробітники цього офісу (домену), але відділ може отримувати доступ до ресурсів в будь-якому офісі (лісі).

```
         ЛІС AD
  ┌────────────────────────────────────────┐
  │  Домен A               Домен B         │
  │  ┌──────────────┐      ┌────────────┐  │
  │  │Global Group: │      │            │  │
  │  │"IT-Kyiv"     │      │ Ресурс:    │  │
  │  │              │      │ FileShare  │  │
  │  │Члени:        │      │            │  │
  │  │✔ user з A    │─────►│ Доступ для │  │
  │  │✔ computer з A│      │ "IT-Kyiv"  │  │
  │  │✗ user з B!   │      │ ✔ дозволено│  │
  │  │  (не можна)  │      └────────────┘  │
  │  └──────────────┘                      │
  └────────────────────────────────────────┘

  Члени — тільки з домену A
  Дозволи — будь-де в лісі (в тому числі в домені B)
```

**Типовий сценарій використання Global Group:**

```
  Задача: Дати бухгалтерам Київського офісу доступ до папки на сервері

  1. Створюємо Global Group "Finance-Kyiv" в Домені A
  2. Додаємо всіх бухгалтерів до "Finance-Kyiv"
  3. Призначаємо "Finance-Kyiv" на папку (через Domain-Local або напряму)

  Новий бухгалтер? → Додай до "Finance-Kyiv" → автоматично отримає доступ
  Звільнили? → Прибери з "Finance-Kyiv" → автоматично втратить доступ
```

> **Простими словами:** Global Group = «відділ». Збирає людей зі спільною роллю в одному домені. Може «ходити» по всьому лісу і отримувати доступ де завгодно.

## Сторінка 33

![Сторінка 33](images/01/page_033.png)

**Ліс AD DS (AD DS Forest)**

- Ліс — це контейнер верхнього рівня в AD DS
- Кожен ліс — це сукупність одного або кількох дерев доменів (domain trees), що мають спільну схему каталогу (directory schema) та спільний глобальний каталог (global catalog)
- Дерево доменів — це сукупність одного або кількох доменів із суміжним простором імен (contiguous namespace)
- Кореневий домен лісу (forest root domain) — перший домен, що створюється при створенні нового лісу
- Кореневий домен лісу містить об'єкти, яких немає в інших доменах лісу:
  - Enterprise Admins (адміністратори підприємства)
  - Schema Admins (адміністратори схеми)
- Ліс може складатися мінімум з одного контролера домену і до необмеженої кількості доменів у кількох деревах доменів

## Сторінка 35

**Важливі характеристики лісу AD DS (Important Characteristics of AD DS Forest)**

- **Межа безпеки (Security boundary).** За замовчуванням жодні користувачі ззовні лісу не можуть отримати доступ до ресурсів усередині лісу. Крім того, усі домени в лісі автоматично довіряють одне одному.

- **Межа реплікації (Replication boundary).** Ліс AD DS є межею реплікації для розділів конфігурації та схеми (configuration and schema partitions) бази даних AD DS. Ліс також є межею реплікації для глобального каталогу (global catalog).

**Організаційні підрозділи (Organizational Units — OUs)**

Організаційний підрозділ (OU) — це контейнерний об'єкт у домені, який можна використовувати для об'єднання користувачів, комп'ютерів, груп та інших об'єктів. До OU можна безпосередньо прив'язувати об'єкти групової політики (Group Policy Objects — GPOs) для управління користувачами та комп'ютерами, що містяться в OU.

## Сторінка 43

![Сторінка 43](images/01/page_043.png)

**Де використовувати OU (Where to Use OUs)**

Дві основні причини для створення OU:

1. **Об'єднання об'єктів для полегшення управління шляхом застосування GPO.** Коли ви призначаєте GPO до OU, параметри застосовуються до всіх об'єктів у OU.

2. **Делегування адміністративного контролю.** Можна призначати дозволи на управління OU, тим самим делегуючи контроль над цим OU користувачеві або групі в AD DS.

OU можна використовувати для відображення ієрархічних логічних структур у вашій організації (відділи, географічні регіони або їх комбінація).

## Сторінка 60

![Сторінка 60](images/01/page_060.png)

**Групова політика (Group Policy)**

**Що таке Group Policy?** Централізований інструмент управління та налаштування комп'ютерів і користувачів. Не потребує додаткового клієнта чи сервера, крім контролера домену (DC).

**Що вона може робити?** Практично все: нескінченна кількість параметрів, розгортання скриптів та управління доступом.

**З чого вона складається?** Служба (GPClient) на кожній машині, Active Directory / Sysvol для GPOs, консоль управління груповою політикою (GPMC).

## Сторінка 61

![Сторінка 61](images/01/page_061.png)

**Визначення, пов'язані з груповою політикою (Definitions Related to Group Policy)**

- **Group Policy** — загальна технологія
- **Group Policy Object (GPO)** — окремий об'єкт, що містить параметри
- **Policy (setting)** — параметр реєстру (registry setting), що не може бути змінений користувачем
- **Group Policy Preferences (GPP)** — надбудова для Group Policy, вбудована в Windows 7
- **preference** — тип параметра, який користувачі можуть змінити

- **Татуювання (Tattooing)** — ефект, коли значення реєстру (registry values), встановлені політикою, залишаються після видалення політики, поки ви явно не видалите їх
- **Параметри без татуювання (Non-tattooed settings)** — видаляються після видалення відповідного параметра групової політики

## Сторінка 136

![Сторінка 136](images/01/page_136.png)

**Що таке WMI? (What Is WMI)**

WMI — реалізація Microsoft стандартів Web-Based Enterprise Management (WBEM) та Common Information Model (CIM). Надає незалежний від платформи засіб збору та передачі інформації про керовані компоненти підприємства (процеси, ключі реєстру, служби, файли тощо).

---

## Сторінка 137

![Сторінка 137](images/01/page_137.png)

**Структура WMI (WMI Structure)**

Чотири компоненти:
1. **WMI providers** — забезпечують доступність керованих ресурсів
2. **CIMOM** — диспетчер трафіку між постачальниками та споживачами WMI
3. **CIM repository** — схема даних; більшість даних ОС знаходиться в `root\cimv2`
4. **WMI scripting library** — інтерфейс для JScript, VBScript, Perl тощо

## Сторінка 139

![Сторінка 139](images/01/page_139.png)

**Запуск програми через WMI (Run a Program Through WMI)**

**VBScript:**
```vb
Const HIDDEN_WINDOW = 0
strComputer = "."
Set objWMIService = GetObject("winmgmts:{impersonationLevel=impersonate}!\\" & strComputer & "\root\cimv2") 
Set objStartup = objWMIService.Get("Win32_ProcessStartup") 
Set objConfig = objStartup.SpawnInstance_
objConfig.ShowWindow = HIDDEN_WINDOW 
Set objProcess = GetObject("winmgmts:root\cimv2:Win32_Process")
errReturn = objProcess.Create("Notepad.exe", null, objConfig, intProcessID)
```

**PowerShell:**
```powershell
$startup=[wmiclass]"Win32_ProcessStartup"
$startup.Properties['ShowWindow'].value=$False 
([wmiclass]"win32_Process").create('notepad.exe','C:\',$Startup)
```

**WMIC:**
```
wmic process call create "c:\windows\notepad.exe"
```
