## Сторінка 14 — Контейнер (Container)

### ❓ Якщо OU краще за контейнер — навіщо взагалі існують контейнери?

Контейнери існують не для адміністратора, а **для самої системи AD**.

Їх створює AD автоматично при розгортанні — адміністратор їх не створює і не обирає:

```
  company.com
  ├── [Контейнер] Builtin      ← створений AD автоматично
  ├── [Контейнер] Computers    ← створений AD автоматично
  ├── [Контейнер] Users        ← створений AD автоматично
  └── [OU] Domain Controllers  ← єдиний вбудований OU
```

| Контейнер | Призначення |
|---|---|
| `Builtin` | Вбудовані локальні групи (Administrators, Users, Guests) |
| `Computers` | Куди автоматично потрапляють нові комп'ютери при вступі в домен |
| `Users` | Куди автоматично потрапляють нові користувачі |

**Реальна проблема контейнерів на практиці:**

```
  Новий ПК приєднався до домену
           ▼
  Потрапив у Computers (контейнер)
           ▼
  GPO не застосовуються!
  Антивірус не встановився,
  налаштування безпеки не застосувались...
           ▼
  Адмін мусить вручну перемістити до OU
```

Саме тому досвідчені адміни одразу переналаштовують куди падають нові об'єкти:
```
redircmp "OU=Workstations,DC=company,DC=com"
redirusr "OU=Users,DC=company,DC=com"
```

**Висновок:** Контейнери не обираєш — вони є і будуть завжди. Завдання адміна — не залишати об'єкти в контейнерах, а одразу переносити їх у відповідні OU.

## Сторінка 17 — Сховище даних (Data Store)

### ❓ Data Store — це сховище файлів `C:\Windows\NTDS\` на Windows Server? Хто має до нього доступ?

**Так, правильно.** Але є важливі уточнення.

**Data Store живе на кожному DC окремо** — не одне централізоване сховище, а копія на кожному сервері що є DC:

```
  DC1 (Київ)              DC2 (Київ)            DC3 (Львів)
  ┌─────────────┐         ┌─────────────┐        ┌─────────────┐
  │ C:\Windows\ │         │ C:\Windows\ │        │ C:\Windows\ │
  │ NTDS\       │◄───────►│ NTDS\       │◄──────►│ NTDS\       │
  │ Ntds.dit    │         │ Ntds.dit    │        │ Ntds.dit    │
  └─────────────┘         └─────────────┘        └─────────────┘
      копія 1                 копія 2                копія 3
```

**Хто має доступ:**

```
  ✔ SYSTEM              ← AD сам читає/пише через службу
  ✔ Domain Admins       ← повний доступ
  ✔ Звичайні юзери      ← читають свої атрибути через AD (але не файл напряму!)
  ✗ Файл Ntds.dit напряму відкрити неможливо — заблокований ОС поки DC працює
```

Доступ до даних відбувається **тільки через AD протоколи** (LDAP, Kerberos) — не напряму до файлу.

**Data Store існує лише на Windows Server з роллю AD DS:**

```
  Звичайний Windows Server  →  встановлення ролі AD DS  →  стає DC
  ┌──────────────────┐                                   ┌──────────────────┐
  │ Немає NTDS папки │    ──────────────────────────►    │ C:\Windows\NTDS\ │
  └──────────────────┘                                   │ Ntds.dit з'явився│
                                                         └──────────────────┘
```

**Важливо — безпека:** Якщо зловмисник скопіює `Ntds.dit` (наприклад, фізично вкраде сервер) — він зможе офлайн витягти **всі хеші паролів** домену. Тому DC завжди тримають у закритих серверних кімнатах, а для філій без фізичного захисту використовують RODC.

## Сторінка 16 — Контролер домену (DC)

### ❓ На одному Windows Server може бути кілька DC (DC1, DC2...)? Чи кожен DC — окремий сервер?

**Кожен DC — це окремий сервер** (фізичний або віртуальний). На одній машині з Windows Server може бути лише один DC.

```
  Фізичний сервер 1          Фізичний сервер 2          Фізичний сервер 3
  ┌─────────────────┐        ┌─────────────────┐        ┌─────────────────┐
  │ Windows Server  │        │ Windows Server  │        │ Windows Server  │
  │ Роль: AD DS     │        │ Роль: AD DS     │        │ Роль: AD DS     │
  │ Ім'я: DC1       │        │ Ім'я: DC2       │        │ Ім'я: DC3       │
  └─────────────────┘        └─────────────────┘        └─────────────────┘
       Київ                       Київ                       Львів
```

**На практиці більшість компаній використовують віртуалізацію.** Тоді на одному фізичному сервері може бути кілька DC — але кожен як окрема віртуальна машина:

```
  Один потужний фізичний сервер (Hyper-V / VMware)
  ┌──────────────────────────────────────────────────┐
  │  Гіпервізор                                      │
  │  ┌────────────┐  ┌────────────┐  ┌────────────┐  │
  │  │   ВМ 1     │  │   ВМ 2     │  │   ВМ 3     │  │
  │  │ Win Server │  │ Win Server │  │ Win Server │  │
  │  │ DC1        │  │ DC2        │  │ File Server│  │
  │  └────────────┘  └────────────┘  └────────────┘  │
  └──────────────────────────────────────────────────┘
```

**Важливе правило при віртуалізації — не тримати всі DC на одному фізичному сервері:**

```
  ПОГАНО ❌                          ДОБРЕ ✔
  ┌──────────────────────┐           ┌──────────┐    ┌──────────┐
  │  Один фіз. сервер    │           │ Сервер 1 │    │ Сервер 2 │
  │  ВМ: DC1             │           │ ВМ: DC1  │    │ ВМ: DC2  │
  │  ВМ: DC2             │           └──────────┘    └──────────┘
  └──────────────────────┘
  Якщо сервер впав —                Якщо Сервер 1 впав —
  обидва DC недоступні!             DC2 на Сервері 2 працює!
```

**Коротко:**
- Один Windows Server = один DC
- DC1, DC2, DC3 — це імена **окремих серверів/ВМ**
- Мінімум **2 DC** на домен (один як резерв)
- У великих компаніях може бути 5-10+ DC розподілених по офісах

## Сторінка 20 — Сайт (Site)

### ❓ Сайт (Site) в AD — це веб-сайт чи щось інше?

Ні, це **не веб-сайт**! Це часта плутанина через однакове слово.

**Сайт в AD DS — це фізичне місце розташування**: офіс, будівля або місто, де знаходяться комп'ютери з швидким мережевим з'єднанням між собою.

```
  "Сайт" в AD ≠ веб-сайт (google.com)
  "Сайт" в AD = фізичний офіс/локація з хорошим LAN-зв'язком
```

**Простий приклад:**

```
  Компанія "Укртех"

  Офіс Київ                        Офіс Львів
  ══════════════                   ══════════════
  AD Сайт: "Kyiv-Site"             AD Сайт: "Lviv-Site"
  Підмережа: 10.1.0.0/24           Підмережа: 10.2.0.0/24
  ┌──────────────────┐             ┌──────────────────┐
  │ DC1, DC2         │             │ DC3              │
  │ 300 комп'ютерів  │◄──WAN──────►│ 50 комп'ютерів   │
  │ 1 Гбіт/с         │  (повільно) │ 1 Гбіт/с         │
  └──────────────────┘             └──────────────────┘
  Всередині сайту — швидко         Між сайтами — повільно
```

**Навіщо AD знає про фізичні локації — щоб не гнати зайвий трафік через WAN:**

| Без сайтів | З сайтами |
|---|---|
| Комп'ютер у Києві може запитати DC у Львові | Комп'ютер у Києві завжди запитує київський DC |
| Реплікація йде постійно через WAN | Реплікація між офісами — за розкладом, стиснена |
| Канал між офісами перевантажений | Канал між офісами вільний |

> **Коротко:** Сайт = фізичний офіс з хорошою мережею всередині. AD використовує сайти щоб оптимізувати трафік між офісами.

## Global Catalog (GC)

### ❓ Коли ми авторизуємося на комп'ютері в AD — він звертається до GC і шукає в якому домені користувач?

Не завжди. GC залучається лише в певних випадках, а авторизацію завжди проводить DC.

```
┌─────────────────────────────────────────────────────────────────┐
│                    ЛОГІН В AD КОМП'ЮТЕР                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                    Як вводить логін?
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
   DOMAIN\username        user@corp.com       user@forest.com
   (SAMAccountName)          (UPN)            (UPN інший домен)
          │                   │                   │
          │                   │                   ▼
          │                   │          ┌─────────────────┐
          │                   │          │  DC свого домену│
          │                   │          │  не знає цей    │
          │                   │          │  UPN → питає GC │
          │                   │          └────────┬────────┘
          │                   │                   │
          │                   │            GC повертає
          │                   │            домен юзера
          │                   │                   │
          └───────────────────┴───────────────────┘
                              │
                              ▼
                   ┌─────────────────────┐
                   │  DC свого домену    │
                   │  (Kerberos AS-REQ)  │
                   │  перевіряє пароль   │
                   └──────────┬──────────┘
                              │
                    Пароль правильний?
                              │
                    ┌─────────┴──────────┐
                   НІ                   ТАК
                    │                    │
                 Відмова                 ▼
                              ┌─────────────────────┐
                              │  DC будує           │
                              │  Access Token       │
                              │  (список груп)      │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  DC питає GC:       │  ← ЗАВЖДИ
                              │  "Universal Groups  │
                              │   цього юзера?"     │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  GC повертає        │
                              │  Universal Groups   │
                              │  з усього лісу      │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  Access Token       │
                              │  повний → TGT       │
                              │  видано             │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  Користувач         │
                              │  увійшов у систему  │
                              └─────────────────────┘
```

GC задіяний при логіні у двох випадках:

| Випадок | Коли | Що робить GC |
|---------|------|--------------|
| UPN логін в multi-domain | `user@other.domain` | Знаходить в якому домені цей UPN |
| Universal Groups | Завжди при логіні | Повертає всі Universal Groups користувача по всьому лісу |

**Що буде якщо GC недоступний:**

```
  DOMAIN\user    →  логін пройде (SAM відомий DC)
  user@corp.com  →  логін пройде (UPN в тому ж домені)
  user@other.com →  ❌ логін не пройде
  Universal Gr.  →  не додадуться в токен → немає доступу до деяких ресурсів
```

**Головне:** GC **не авторизує** — це робить DC. GC потрібен щоб знайти домен при UPN логіні та додати Universal Groups в Access Token.

---

### ❓ UPN — це те саме що VPN?

Ні, це абсолютно різні речі.

**UPN (User Principal Name)** — формат імені користувача в AD, виглядає як email:

```
┌─────────────────────────────────────────────────┐
│  Один і той самий користувач — два формати:     │
│                                                 │
│  SAMAccountName:  COMPANY\ivan.petrenko         │
│  UPN:             ivan.petrenko@company.com     │
└─────────────────────────────────────────────────┘
```

**VPN (Virtual Private Network)** — технологія шифрованого тунелю до мережі. З AD напряму не пов'язана.

Чому UPN зручніший за SAMAccountName:

```
  COMPANY\ivan.petrenko    ← треба знати назву домену
  ivan.petrenko@company.com  ← схожий на email, легше запам'ятати
```

---

### ❓ Коли я авторизуюся як `ivan.petrenko@company.com` на комп'ютері який в іншому домені але в спільному лісі — він звертається до GC щоб знайти в якому домені цей користувач?

Так, саме так:

```
┌─────────────────────────────────────────────────────────────────┐
│  ЛІЕС (forest.com)                                              │
│                                                                 │
│         ┌─────────────────┐                                     │
│         │  forest.com     │  ← батьківський домен (GC тут)     │
│         └────────┬────────┘                                     │
│                  │                                              │
│        ┌─────────┴──────────┐                                   │
│        │                    │                                   │
│  ┌─────┴──────┐      ┌──────┴─────┐                            │
│  │  kyiv.     │      │  lviv.     │                            │
│  │  forest.com│      │  forest.com│                            │
│  │            │      │            │                            │
│  │ 💻 твій    │      │ 👤 ivan    │                            │
│  │  комп'ютер │      │  petrenko  │                            │
│  └────────────┘      └────────────┘                            │
└─────────────────────────────────────────────────────────────────┘
```

```
Ти вводиш: ivan.petrenko@forest.com
                    │
                    ▼
        DC домену kyiv.forest.com
        "Хто такий @forest.com? Це не мій домен..."
                    │
                    ▼
        Запит до GC (forest.com)
        "Де живе ivan.petrenko?"
                    │
                    ▼
        GC відповідає:
        "ivan.petrenko є в lviv.forest.com"
                    │
                    ▼
        DC kyiv звертається до DC lviv
        для перевірки пароля
                    │
                    ▼
        Пароль вірний → Access Token → логін успішний
```

GC знає **де** шукати користувача, але повну перевірку пароля робить рідний DC користувача — `lviv.forest.com`.

## Сторінка 23 — Managed Service Accounts (MSA)

### ❓ Яка реальна ціль MSA і які роботи він виконує?

Спочатку — проблема яку MSA вирішує:

```
БЕЗ MSA — звичайний акаунт для сервісу:

  SQL Server запускається від імені:  svc_sql / Password123!
                                              │
                    ┌─────────────────────────┼──────────────────────┐
                    │                         │                      │
            Пароль не змінюється      Адмін знає пароль      Пароль прописаний
            роками (бо якщо           і може увійти          вручну скрізь
            змінити — сервіс          куди завгодно
            впаде)
```

З MSA — AD сам керує паролем, ніхто його не знає:

```
  ЗА MSA:

  SQL Server запускається від імені:  svc_sql$
                                           │
                    ┌──────────────────────┼─────────────────────┐
                    │                      │                      │
            Пароль змінюється       Ніхто не знає          Прив'язаний
            автоматично             пароль —                до одного
            кожні 30 днів          навіть адмін             комп'ютера
```

**Реальні сервіси де використовується MSA:**

```
  ✔ SQL Server          — база даних
  ✔ IIS App Pool        — веб-додаток
  ✔ Windows Service     — фоновий сервіс
  ✔ Scheduled Task      — заплановане завдання
  ✔ Exchange Server     — поштовий сервер
```

**Два види MSA:**

| | sMSA (standalone) | gMSA (group) |
|---|---|---|
| Де працює | Один сервер | Кілька серверів |
| Для чого | Звичайний сервіс | Кластер, ферма, load balancer |

---

### ❓ Додаток підключається до SQL через MSA — він автоматично вписується в середовище?

Не зовсім. MSA використовує **не додаток**, а **сам сервіс SQL Server** при запуску в Windows.

```
  Користувач
      │  (відкриває додаток)
      ▼
  ┌─────────────┐
  │   Додаток   │  ← працює від імені користувача або свого акаунту
  └──────┬──────┘
         │  (запит до бази)
         ▼
  ┌─────────────┐
  │  SQL Server │  ← ось тут MSA!
  │  svc_sql$   │    SQL Server запускається від імені MSA
  └──────┬──────┘    і має права читати/писати в БД
         │
         ▼
  ┌─────────────┐
  │   База      │
  │   Даних     │
  └─────────────┘
```

MSA прописується в Windows Services при запуску сервісу:

```
  Windows Services (services.msc)

  ┌──────────────────────────────────────────────┐
  │ SQL Server (MSSQLSERVER)                     │
  │ Status:  Running                             │
  │ Log on:  COMPANY\svc_sql$   ← ось MSA       │
  └──────────────────────────────────────────────┘
```

Windows сам іде до AD, отримує пароль і запускає сервіс — ні адмін ні додаток цей пароль не бачать.

**Коротко — різниця:**

```
  Додаток    ──► підключається до SQL  ──► використовує свій акаунт
  SQL Server ──► запускається в Windows ──► використовує MSA
```

MSA — це не про підключення до БД, це про **запуск самого сервісу** в операційній системі.

## Сторінка 26 — Групи (Security vs Distribution)

### ❓ В чому різниця між Security Group і Distribution Group?

Різниця в одному — **чи можна призначити права доступу**.

```
┌─────────────────────────────────────────────────────────────────┐
│              Security Group      │    Distribution Group        │
├─────────────────────────────────────────────────────────────────┤
│  Права доступу (ACL)    ✔        │         ✗                    │
│  Поштова розсилка       ✔        │         ✔                    │
│  GPO фільтрація         ✔        │         ✗                    │
└─────────────────────────────────────────────────────────────────┘
```

Єдина реальна причина використовувати Distribution Group — якщо **навмисно** хочеш групу яка ніколи не матиме прав. Наприклад список розсилки із зовнішніми партнерами — фізично неможливо випадково призначити їй права на файловий сервер.

**Порада:** Завжди створюй Security Groups — вони вміють все що Distribution Group + ще й безпеку.

## Сторінка 28 — Local Group

### ❓ Що дає Local Group?

Local Group дає права **тільки на тому комп'ютері де вона створена**.

```
  Local Group = інструмент всередині Windows

  Все що відбувається на цьому ПК → Local Group працює
  Все що йде по мережі з інших ПК → Local Group не працює
```

Як кімната з замком — хто вже всередині (залогінився, підключився по RDP) той підпадає під правила. Хто ззовні по мережі — замок його не бачить.

**Де реально використовується:**

```
  ✔ Локальні адміністратори ПК
    Administrators (вбудована local group)
    → хто може керувати саме цим комп'ютером

  ✔ Локальні користувачі ПК
    Users, Guests (вбудовані local groups)
    → хто може входити в систему локально

  ✔ Ізольований комп'ютер без домену
    Домашній ПК, окремий сервер — де немає AD взагалі
```

---

### ❓ Є розшарована папка на RDP сервері, користувачі ivan, katya, nastya підключені по RDP — вони матимуть доступ до папки?

Так, але залежить від прав на папку.

```
  RDP Сервер (один комп'ютер)
  ┌─────────────────────────────────────────────────┐
  │  Локальні користувачі:                          │
  │    👤 ivan   ← підключений по RDP               │
  │    👤 katya  ← підключена по RDP                │
  │    👤 nastya ← підключена по RDP                │
  │                                                 │
  │  📁 C:\SharedFolder                             │
  └─────────────────────────────────────────────────┘
```

Всі троє на **одній машині** — тому Local Group тут працює:

```
  Варіант 1 — папка відкрита для всіх:
  Local Group "Users" → має доступ → ivan, katya, nastya бачать папку ✔

  Варіант 2 — папка тільки для деяких:
  Local Group "SharedAccess" → додаєш тільки ivan і katya
  → nastya не бачить папку ✗
```

**Важливо — розшарована папка і локальна папка це різне:**

```
  📁 Локальна (C:\SharedFolder)
  → бачать тільки ті хто підключений до цього ПК
  → Local Group достатньо

  📁 Розшарована по мережі (\\server\SharedFolder)
  → доступна з ІНШИХ комп'ютерів
  → потрібна Domain Security Group
```

## Сторінка 29 — Domain-Local Group і AGDLP

### ❓ Яку групу краще використовувати і як вони пов'язані між собою?

На практиці є одна проста схема — **AGDLP**:

```
  Account → Global → Domain-Local → Permission

  👤 ivan.petrenko          ← користувач (Account)
       │
       └─► Global Group                ← збираєш відділ
           "Бухгалтерія"                 (хто є хто)
                │
                └─► Domain-Local Group  ← даєш доступ до ресурсу
                    "Доступ_Фінанси"      (до чого)
                         │
                         └─► 📁 Папка Finance   ← права (Permission)
```

```
┌──────────────────┬────────────────────────────────────────────┐
│  Група           │  Просте пояснення                          │
├──────────────────┼────────────────────────────────────────────┤
│  Local Group     │  Тільки для одного ПК                      │
│  Domain-Local    │  Для ресурсів домену (папки, принтери)     │
│  Global Group    │  Для збирання користувачів по відділах     │
│  Universal       │  Для великих організацій з кількох лісів  │
└──────────────────┴────────────────────────────────────────────┘
```

- **Global** — відповідає на питання **хто** (відділ, роль)
- **Domain-Local** — відповідає на питання **до чого** (папка, принтер)
