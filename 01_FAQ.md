## Сторінка 14 — Контейнер (Container)

### ❓ Якщо OU краще за контейнер — навіщо взагалі існують контейнери?

Контейнери існують не для адміністратора, а **для самої системи AD**.

Їх створює AD автоматично при розгортанні — адміністратор їх не створює і не обирає:

```
  company.com
  ├── [Контейнер] Builtin      ← створений AD автоматично
  ├── [Контейнер] Computers    ← створений AD автоматично
  ├── [Контейнер] Users        ← створений AD автоматично
  └── [OU] Domain Controllers  ← єдиний вбудований OU
```

| Контейнер | Призначення |
|---|---|
| `Builtin` | Вбудовані локальні групи (Administrators, Users, Guests) |
| `Computers` | Куди автоматично потрапляють нові комп'ютери при вступі в домен |
| `Users` | Куди автоматично потрапляють нові користувачі |

**Реальна проблема контейнерів на практиці:**

```
  Новий ПК приєднався до домену
           ▼
  Потрапив у Computers (контейнер)
           ▼
  GPO не застосовуються!
  Антивірус не встановився,
  налаштування безпеки не застосувались...
           ▼
  Адмін мусить вручну перемістити до OU
```

Саме тому досвідчені адміни одразу переналаштовують куди падають нові об'єкти:
```
redircmp "OU=Workstations,DC=company,DC=com"
redirusr "OU=Users,DC=company,DC=com"
```

**Висновок:** Контейнери не обираєш — вони є і будуть завжди. Завдання адміна — не залишати об'єкти в контейнерах, а одразу переносити їх у відповідні OU.

## Сторінка 17 — Сховище даних (Data Store)

### ❓ Data Store — це сховище файлів `C:\Windows\NTDS\` на Windows Server? Хто має до нього доступ?

**Так, правильно.** Але є важливі уточнення.

**Data Store живе на кожному DC окремо** — не одне централізоване сховище, а копія на кожному сервері що є DC:

```
  DC1 (Київ)              DC2 (Київ)            DC3 (Львів)
  ┌─────────────┐         ┌─────────────┐        ┌─────────────┐
  │ C:\Windows\ │         │ C:\Windows\ │        │ C:\Windows\ │
  │ NTDS\       │◄───────►│ NTDS\       │◄──────►│ NTDS\       │
  │ Ntds.dit    │         │ Ntds.dit    │        │ Ntds.dit    │
  └─────────────┘         └─────────────┘        └─────────────┘
      копія 1                 копія 2                копія 3
```

**Хто має доступ:**

```
  ✔ SYSTEM              ← AD сам читає/пише через службу
  ✔ Domain Admins       ← повний доступ
  ✔ Звичайні юзери      ← читають свої атрибути через AD (але не файл напряму!)
  ✗ Файл Ntds.dit напряму відкрити неможливо — заблокований ОС поки DC працює
```

Доступ до даних відбувається **тільки через AD протоколи** (LDAP, Kerberos) — не напряму до файлу.

**Data Store існує лише на Windows Server з роллю AD DS:**

```
  Звичайний Windows Server  →  встановлення ролі AD DS  →  стає DC
  ┌──────────────────┐                                   ┌──────────────────┐
  │ Немає NTDS папки │    ──────────────────────────►    │ C:\Windows\NTDS\ │
  └──────────────────┘                                   │ Ntds.dit з'явився│
                                                         └──────────────────┘
```

**Важливо — безпека:** Якщо зловмисник скопіює `Ntds.dit` (наприклад, фізично вкраде сервер) — він зможе офлайн витягти **всі хеші паролів** домену. Тому DC завжди тримають у закритих серверних кімнатах, а для філій без фізичного захисту використовують RODC.

## Сторінка 16 — Контролер домену (DC)

### ❓ На одному Windows Server може бути кілька DC (DC1, DC2...)? Чи кожен DC — окремий сервер?

**Кожен DC — це окремий сервер** (фізичний або віртуальний). На одній машині з Windows Server може бути лише один DC.

```
  Фізичний сервер 1          Фізичний сервер 2          Фізичний сервер 3
  ┌─────────────────┐        ┌─────────────────┐        ┌─────────────────┐
  │ Windows Server  │        │ Windows Server  │        │ Windows Server  │
  │ Роль: AD DS     │        │ Роль: AD DS     │        │ Роль: AD DS     │
  │ Ім'я: DC1       │        │ Ім'я: DC2       │        │ Ім'я: DC3       │
  └─────────────────┘        └─────────────────┘        └─────────────────┘
       Київ                       Київ                       Львів
```

**На практиці більшість компаній використовують віртуалізацію.** Тоді на одному фізичному сервері може бути кілька DC — але кожен як окрема віртуальна машина:

```
  Один потужний фізичний сервер (Hyper-V / VMware)
  ┌──────────────────────────────────────────────────┐
  │  Гіпервізор                                      │
  │  ┌────────────┐  ┌────────────┐  ┌────────────┐  │
  │  │   ВМ 1     │  │   ВМ 2     │  │   ВМ 3     │  │
  │  │ Win Server │  │ Win Server │  │ Win Server │  │
  │  │ DC1        │  │ DC2        │  │ File Server│  │
  │  └────────────┘  └────────────┘  └────────────┘  │
  └──────────────────────────────────────────────────┘
```

**Важливе правило при віртуалізації — не тримати всі DC на одному фізичному сервері:**

```
  ПОГАНО ❌                          ДОБРЕ ✔
  ┌──────────────────────┐           ┌──────────┐    ┌──────────┐
  │  Один фіз. сервер    │           │ Сервер 1 │    │ Сервер 2 │
  │  ВМ: DC1             │           │ ВМ: DC1  │    │ ВМ: DC2  │
  │  ВМ: DC2             │           └──────────┘    └──────────┘
  └──────────────────────┘
  Якщо сервер впав —                Якщо Сервер 1 впав —
  обидва DC недоступні!             DC2 на Сервері 2 працює!
```

**Коротко:**
- Один Windows Server = один DC
- DC1, DC2, DC3 — це імена **окремих серверів/ВМ**
- Мінімум **2 DC** на домен (один як резерв)
- У великих компаніях може бути 5-10+ DC розподілених по офісах

## Global Catalog (GC)

### ❓ Коли ми авторизуємося на комп'ютері в AD — він звертається до GC і шукає в якому домені користувач?

Не завжди. GC залучається лише в певних випадках, а авторизацію завжди проводить DC.

```
┌─────────────────────────────────────────────────────────────────┐
│                    ЛОГІН В AD КОМП'ЮТЕР                         │
└─────────────────────────────────────────────────────────────────┘
                              │
                    Як вводить логін?
                              │
          ┌───────────────────┼───────────────────┐
          │                   │                   │
   DOMAIN\username        user@corp.com       user@forest.com
   (SAMAccountName)          (UPN)            (UPN інший домен)
          │                   │                   │
          │                   │                   ▼
          │                   │          ┌─────────────────┐
          │                   │          │  DC свого домену│
          │                   │          │  не знає цей    │
          │                   │          │  UPN → питає GC │
          │                   │          └────────┬────────┘
          │                   │                   │
          │                   │            GC повертає
          │                   │            домен юзера
          │                   │                   │
          └───────────────────┴───────────────────┘
                              │
                              ▼
                   ┌─────────────────────┐
                   │  DC свого домену    │
                   │  (Kerberos AS-REQ)  │
                   │  перевіряє пароль   │
                   └──────────┬──────────┘
                              │
                    Пароль правильний?
                              │
                    ┌─────────┴──────────┐
                   НІ                   ТАК
                    │                    │
                 Відмова                 ▼
                              ┌─────────────────────┐
                              │  DC будує           │
                              │  Access Token       │
                              │  (список груп)      │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  DC питає GC:       │  ← ЗАВЖДИ
                              │  "Universal Groups  │
                              │   цього юзера?"     │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  GC повертає        │
                              │  Universal Groups   │
                              │  з усього лісу      │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  Access Token       │
                              │  повний → TGT       │
                              │  видано             │
                              └──────────┬──────────┘
                                         │
                              ┌──────────▼──────────┐
                              │  Користувач         │
                              │  увійшов у систему  │
                              └─────────────────────┘
```

GC задіяний при логіні у двох випадках:

| Випадок | Коли | Що робить GC |
|---------|------|--------------|
| UPN логін в multi-domain | `user@other.domain` | Знаходить в якому домені цей UPN |
| Universal Groups | Завжди при логіні | Повертає всі Universal Groups користувача по всьому лісу |

**Що буде якщо GC недоступний:**

```
  DOMAIN\user    →  логін пройде (SAM відомий DC)
  user@corp.com  →  логін пройде (UPN в тому ж домені)
  user@other.com →  ❌ логін не пройде
  Universal Gr.  →  не додадуться в токен → немає доступу до деяких ресурсів
```

**Головне:** GC **не авторизує** — це робить DC. GC потрібен щоб знайти домен при UPN логіні та додати Universal Groups в Access Token.

### ❓ UPN — це те саме що VPN?

Ні, це абсолютно різні речі.

**UPN (User Principal Name)** — формат імені користувача в AD, виглядає як email:

```
┌─────────────────────────────────────────────────┐
│  Один і той самий користувач — два формати:     │
│                                                 │
│  SAMAccountName:  COMPANY\ivan.petrenko         │
│  UPN:             ivan.petrenko@company.com     │
└─────────────────────────────────────────────────┘
```

**VPN (Virtual Private Network)** — технологія шифрованого тунелю до мережі. З AD напряму не пов'язана.

Чому UPN зручніший за SAMAccountName:

```
  COMPANY\ivan.petrenko    ← треба знати назву домену
  ivan.petrenko@company.com  ← схожий на email, легше запам'ятати
```

### ❓ Коли я авторизуюся як `ivan.petrenko@company.com` на комп'ютері який в іншому домені але в спільному лісі — він звертається до GC щоб знайти в якому домені цей користувач?

Так, саме так:

```
┌─────────────────────────────────────────────────────────────────┐
│  ЛІЕС (forest.com)                                              │
│                                                                 │
│         ┌─────────────────┐                                     │
│         │  forest.com     │  ← батьківський домен (GC тут)     │
│         └────────┬────────┘                                     │
│                  │                                              │
│        ┌─────────┴──────────┐                                   │
│        │                    │                                   │
│  ┌─────┴──────┐      ┌──────┴─────┐                            │
│  │  kyiv.     │      │  lviv.     │                            │
│  │  forest.com│      │  forest.com│                            │
│  │            │      │            │                            │
│  │ 💻 твій    │      │ 👤 ivan    │                            │
│  │  комп'ютер │      │  petrenko  │                            │
│  └────────────┘      └────────────┘                            │
└─────────────────────────────────────────────────────────────────┘
```

```
Ти вводиш: ivan.petrenko@forest.com
                    │
                    ▼
        DC домену kyiv.forest.com
        "Хто такий @forest.com? Це не мій домен..."
                    │
                    ▼
        Запит до GC (forest.com)
        "Де живе ivan.petrenko?"
                    │
                    ▼
        GC відповідає:
        "ivan.petrenko є в lviv.forest.com"
                    │
                    ▼
        DC kyiv звертається до DC lviv
        для перевірки пароля
                    │
                    ▼
        Пароль вірний → Access Token → логін успішний
```

GC знає **де** шукати користувача, але повну перевірку пароля робить рідний DC користувача — `lviv.forest.com`.
